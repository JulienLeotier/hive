## What Went Well

### 1. Architecture-First Approach Paid Off
The BMAD method's structure (brainstorm → brief → PRD → architecture → epics → implement) meant every line of code had a clear purpose tracing back to a requirement. Zero wasted implementation effort.

### 2. Go + SQLite = Zero-Dependency Bliss
The choice of Go with embedded SQLite (modernc.org/sqlite, pure Go) delivered on the "single binary, zero dependencies" promise. No CGO, no external DB, no runtime dependencies. Binary cross-compiles cleanly.

### 3. Adapter Protocol Design Is Clean
The 5-method `Adapter` interface (Declare, Invoke, Health, Checkpoint, Resume) is minimal enough to implement in <20 lines but expressive enough for real orchestration. The compile-time interface verification (`var _ Adapter = (*HTTPAdapter)(nil)`) catches protocol violations at build time.

### 4. Event Bus Architecture Is Solid
In-process event bus with SQLite persistence, prefix-based pub/sub, and ordered delivery. The `safeCall` panic recovery (added in code review) prevents subscriber failures from crashing the system.

### 5. Code Review Caught Real Bugs
The adversarial code review (Blind Hunter + Edge Case Hunter) found 14 real issues including:
- Command injection vulnerability (HIGH)
- O(N) bcrypt DoS vector (HIGH)
- Double-close panic (HIGH)
- Phantom state transition events (HIGH)

All fixed. This validates the BMAD code-review-before-ship workflow.

## What We Learned

### 1. State Machine Transitions Need RowsAffected Checks
Every SQL UPDATE that transitions state MUST check `RowsAffected()`. Without this, phantom events are published for transitions that never happened. This was the most widespread bug (affected task.Start, task.Complete, task.Fail, workflow.UpdateStatus).

**Action:** Establish as a coding standard — all state-transition UPDATEs must check RowsAffected.

### 2. crypto/rand Everywhere, math/rand Nowhere
The agent manager used `math/rand` for ULID generation while task and workflow used `crypto/rand`. Inconsistency creates predictable IDs.

**Action:** Lint rule or code review checklist item: never use `math/rand` for IDs or security-sensitive values.

### 3. Fail-Closed on Auth Uncertainty
`HasKeys()` originally returned `false` on DB error, disabling auth entirely. The fix returns `true` (require auth) when uncertain.

**Action:** Default security posture should always be fail-closed. Document this principle.

### 4. Response Body Limits Are Essential
`io.ReadAll` without `LimitReader` is an OOM vector. Every HTTP client reading external responses needs a size limit.

**Action:** Add `io.LimitReader` to the coding standards checklist.

### 5. BMAD Method Scales Well for Solo Dev
The full BMAD workflow (brainstorming → brief → PRD → architecture → epics → sprint → dev → review → retro) worked effectively even for a single developer. The structured approach prevented scope creep and ensured nothing was missed.

## Challenges

### 1. Story Granularity Varied
Some stories were naturally atomic (Story 1.1: bootstrap) while others bundled multiple FRs (Story 2.1: event bus covered 5 FRs). Consistent story sizing would improve predictability.

### 2. Tests Could Be More Comprehensive
Current tests cover happy paths and basic error cases. Edge cases like concurrent access, large payloads, and malformed inputs are under-tested. The code review caught what tests missed.

### 3. No Integration Tests Yet
All tests are unit-level (package-scoped). No end-to-end test exercises the full flow: register agent → create workflow → run → get results. This should be priority for v0.2.
